# OCP Scenario Expansion

Create CI config expansions for new OCP versions in the `openshift/release` repository. Each expansion copies an existing config for OCP version N and creates a new one for OCP version N+1, updating version references and Jira metadata.

## When to Use This Skill

Use this when:
- A new OCP version needs interop CI testing (e.g., 4.22 → 4.23)
- Jira epic assigns FIPS and/or Component Readiness expansion tickets
- Need to create `ci-operator/config/` YAML files for new OCP version
- Want to run the expansion as an automated subagent workflow

Depends on: @../openshift-release-ci/SKILL.md for CI patterns. For parallel subagent execution, rely on an **external** workflow skill (for example a global `agentic-workflows` skill); that skill is **not** bundled in this plugin.

## Expansion Types

Each product gets two expansion types per OCP version:

| Aspect | Component Readiness (CR) | FIPS |
|--------|-------------------------|------|
| Workflow | `firewatch-ipi-aws-cr` | `firewatch-ipi-aws` |
| `FIPS_ENABLED` | not set | `"true"` |
| `FIREWATCH_FAIL_WITH_TEST_FAILURES` | not set | `"true"` |
| `FIREWATCH_CONFIG_FILE_PATH` | not set | `aws-ipi/lp-interop.json` |
| Jira project | product-specific (ROX, GITOPS) | `LPINTEROP` |
| Jira labels | `"N.NN-lp-cr"` | `"N.NN-lp", "fips"` |
| Cron | `0 3,15 * * *` (active) | `0 23 31 2 *` (disabled) |
| Additional env | `MAP_TESTS`, `OCP_VERSION`, `REPORTPORTAL_CMP` | none |

## Version Edit Checklist

Fields to update when copying a config from version N to N+1:

### Always Change

| Field Path | Example Change |
|-----------|----------------|
| `releases.latest.candidate.version` | `"4.21"` → `"4.22"` |
| `env.FIREWATCH_DEFAULT_JIRA_ADDITIONAL_LABELS` | `"4.21-lp"` → `"4.22-lp"` |

### CR-Specific

| Field Path | Example Change |
|-----------|----------------|
| `env.OCP_VERSION` | `"4.21"` → `"4.22"` |

### FIPS-Specific (add these)

| Field Path | Value |
|-----------|-------|
| `env.FIREWATCH_FAIL_WITH_TEST_FAILURES` | `"true"` |

### Never Change Manually

| Field | Reason |
|-------|--------|
| `zz_generated_metadata` | Auto-generated by `make update` from filename |
| `base_images.cli.name` for CR configs | Uses fixed older CLI version (e.g., `"4.14"`) |

### Product Version Bumps

Some products bump their version alongside OCP (e.g., GitOps v1.18 → v1.19). The branch ref changes via the **filename** — rename the file from `v1.18` to `v1.19` in the basename, and `make update` sets `zz_generated_metadata.branch` accordingly.

## Naming Conventions

### Config File Patterns

| Type | Pattern |
|------|---------|
| CR | `<org>-<repo>-<branch>__<product>-ocp-N.NN-lp-interop-cr.yaml` |
| FIPS | `<org>-<repo>-<branch>__<product>-ocpN.NN-lp-interop-fips.yaml` |

Note the subtle difference: CR uses `-ocp-` (hyphen before ocp), FIPS uses `-ocp` (no hyphen) in some products. Follow the existing convention for each product.

### FIPS `-fips` Suffix Rule

For single-test FIPS configs (starting with 4.22):
1. Append `-fips` to the config filename (sets the variant)
2. Keep `-fips` in the `tests[].as` field to match `.config.prowgen` entries

### Multi-Test Config Exception

Configs with multiple tests (e.g., Quay with both FIPS and non-FIPS entries) cannot use the filename `-fips` approach — there's only one variant per config file. Keep `-fips` in the `tests[].as` field only.

### Test Name Limits

| Limit | Value |
|-------|-------|
| `tests[].as` field | 61 characters max |
| Full Prow Job Run Name | 63 characters (K8s label limit) |

Exceeding the limit causes truncation with a hash suffix, breaking telemetry. Context like OCP version and product name is already in the variant from the filename — keep test names concise.

## Implementation Workflow

### Per-Product Pipeline

```
1. Create branch from upstream/main
2. For each expansion type (FIPS, CR):
   a. Copy source config → target path
   b. Apply version edits (StrReplace)
   c. Stage files
3. Run: SKIP_PULL=true make ci-operator-config
4. Run: SKIP_PULL=true make jobs
5. Commit all configs + generated periodics
6. Push branch to fork
7. Create PR to openshift/release
8. Comment on Jira tickets with PR link
9. Transition Jira tickets to In Progress
```

### Parallelism

Products are fully independent — different directories, different branches. Run all products in parallel. Within a product, FIPS and CR share a branch and `make update` run, so they're sequential.

### Pre-flight Checks

Before starting:
- Verify fork exists (or create via `GitHub-fork_repository`)
- For products with version bumps, verify target branch exists (e.g., `v1.19` in `redhat-developer/gitops-operator`)
- Check `.config.prowgen` for existing job name entries (usually no update needed)

## Subagent Workflow

For automated execution, use the workflow definitions in `workflow/`:

```
workflow/
  workflow.yaml              # 3-phase orchestration
  subagents/
    repo-setup.yaml          # Clone/fork/branch prerequisites
    product-expander.yaml    # Config creation + make update
    pr-lifecycle.yaml        # PR + Jira + tracking
```

Input is a declarative product config (see `assets/product-config-example.yaml`).

## FIREWATCH Configuration Patterns

### CR Config

```yaml
FIREWATCH_CONFIG: |
  {
    "failure_rules": [
      {"step": "<product>-*", "failure_type": "all", "classification": "Test Failure",
       "jira_assignee": "!default", "jira_additional_labels": ["!default","interop-tests"]}
    ]
  }
FIREWATCH_DEFAULT_JIRA_PROJECT: <PRODUCT_PROJECT>
```

### FIPS Config

```yaml
FIREWATCH_CONFIG: |
  {
    "failure_rules": [
      {"step": "<product>-*", "failure_type": "pod_failure", "classification": "Test Execution",
       "jira_additional_labels": ["!default","interop-tests"]},
      {"step": "<product>-*", "failure_type": "test_failure", "classification": "Test Failure",
       "jira_project": "<PRODUCT_PROJECT>", "jira_assignee": "!default",
       "jira_additional_labels": ["!default","interop-tests"]}
    ]
  }
FIREWATCH_CONFIG_FILE_PATH: https://raw.githubusercontent.com/CSPI-QE/cspi-utils/main/firewatch-base-configs/aws-ipi/lp-interop.json
FIREWATCH_DEFAULT_JIRA_PROJECT: LPINTEROP
FIREWATCH_FAIL_WITH_TEST_FAILURES: "true"
```
