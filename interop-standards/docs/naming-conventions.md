# OCP 4.22 Naming Conventions

Conventions established by the team for OCP 4.22 expansions. These differ from earlier versions.

## Config File Naming

### Component Readiness (non-FIPS)

Pattern: `<org>-<repo>-<branch>__<product>-ocp-4.22-lp-interop-cr.yaml`

The `-ocp-4.22` uses a hyphen before "ocp" (not `ocp4.22`).

### FIPS (new convention for 4.22+)

Pattern: `<org>-<repo>-<branch>__<product>-ocp4.22-lp-interop-fips.yaml`

Starting with OCP 4.22, FIPS-only configs append `-fips` to the YAML filename.

Example transformation:
```
4.21: red-hat-storage-ocs-ci-master__odf-ocp4.21-lp-interop.yaml
4.22: red-hat-storage-ocs-ci-master__odf-ocp4.22-lp-interop-fips.yaml
```

### Multi-Test Config Exception

Config files containing multiple tests (including FIPS) do NOT get renamed. The `-fips` suffix cannot be appended to the variant because there is only one `zz_generated_metadata.variant` per config file — appending `-fips` would negate the non-FIPS tests in that config. Example: `quay-quay-tests-master__ocp-421-quay.yaml`.

For these multi-test configs, `-fips` remains in the `tests[].as` field for the FIPS test entry.

## `zz_generated_metadata` Block

Do NOT manually edit the `zz_generated_metadata` block. It is auto-generated by `make update` based on the config filename. The variant value is derived from the portion after `__` in the filename.

## Test Name (`tests[].as` field)

The `as` field has a hard limit of 61 characters.

### Avoid Redundancy

Do NOT include terms in the `as` field that are already part of the variant, repository, or organization name. The CI Operator concatenates the Prow job name from: `job-type-prefix + org + repo + branch + variant + as-field`.

Bad: `odf-interop-aws-fips` (duplicates "fips" already in variant)
Good: `odf-interop-aws`

### FIPS Indication

For single-test FIPS configs, instead of putting `-fips` in the `as` field:
1. Remove `-fips` from `tests:as:` field
2. Append `-fips` to `*.yaml` config filename (which sets the variant automatically)

This keeps job names under the 61-char limit and avoids duplicate tokens like:
`periodic-ci-red-hat-storage-ocs-ci-master-odf-ocp4.22-lp-interop-fips-odf-interop-aws-fips`

### Job Run Name Length

The full Prow Job Run Name follows the pattern:
`periodic-ci-<org>-<repo>-<branch>-<variant>-<testName>`

Limits to watch:
- `tests[].as` (test name): 61-character hard limit; `make update` fails if exceeded
- Kubernetes label values: 63-character limit; exceeded names get truncated with a hash suffix, which breaks telemetry and makes debugging impractical

Keep test names concise — context like the OCP version and product name is already encoded in the variant (from the filename).

## CR Variant Detection

**Updated 2026-02-26** — Edo confirmed with Forrest (TRT) that Component Readiness detection does NOT require the `-cr` marker to be in the CI Operator variant (config filename). CR parses the full Job Run Name to identify its variant classification.

Source: [sippy variant registry](https://github.com/openshift/sippy/blob/6ad73ae236bacbc841b30042ad576cc64038f4be/pkg/variantregistry/ocp.go#L1138)

What this means:
- `-cr` can appear in the test name (`tests[].as`) instead of the filename
- Example: `as: cr-acs-tests-aws` in a config without `-cr` in filename
- CR and FIPS tests could theoretically share a single config file

**Current approach (4.22 sprint)**: Separate config files with `-cr` in filename, matching the 4.21 pattern. The shared-config optimization is a potential future improvement once the team formally adopts it.

**Future optimization**: Remove `-cr` from config filenames, merge CR and non-FIPS tests into shared config files, use `cr-` prefix in test names. This reduces maintenance burden (fewer config files per product).

## Release Value in Filename

Use `-ocp-4.22` (with hyphen separator), NOT `-ocp4.21` (no separator). The release value should be separated from the version prefix.

## OCP_VERSION Environment Variable

Always include `OCP_VERSION: "4.22"` in the config env. This ensures the `4.22` jobs are correctly aligned.
