# OCP Scenario Expansion Process Guide

Step-by-step process for creating OCP version expansions in `openshift/release` CI configs. Extracted from INTEROP-8881 team discussion and reference PRs.

## Overview

An "expansion" copies an existing CI config for OCP version N and creates a new one for OCP version N+1. The config defines periodic Prow jobs that run layered product interop tests against a new OCP version.

Two types of expansions per product:
- **Component Readiness (CR)**: Non-FIPS tests, workflow `firewatch-ipi-aws-cr`
- **FIPS**: FIPS-enabled tests, workflow `firewatch-ipi-aws`, with `FIPS_ENABLED: "true"`

## Step-by-Step Process

### 1. Identify source config

Find the existing 4.21 config for your product:

```bash
ls ci-operator/config/<org>/<repo>/*4.21*
```

### 2. Copy and rename

For Component Readiness:
```bash
cp <source>__<variant>-ocp-4.21-lp-interop-cr.yaml \
   <source>__<variant>-ocp-4.22-lp-interop-cr.yaml
```

For FIPS (new naming convention starting with 4.22):
```bash
cp <source>__<variant>-ocp4.21-lp-interop.yaml \
   <source>__<variant>-ocp4.22-lp-interop-fips.yaml
```

### 3. Update version references

In the new file, change:

| Field Path | Change |
|-----------|--------|
| `releases.latest.candidate.version` | `"4.21"` ŌåÆ `"4.22"` |
| `env.OCP_VERSION` | `"4.21"` ŌåÆ `"4.22"` |
| `env.FIREWATCH_DEFAULT_JIRA_ADDITIONAL_LABELS` | `"4.21-lp"` ŌåÆ `"4.22-lp"` |

Do NOT manually edit `zz_generated_metadata`. It is auto-generated by `make update` from the config filename. The variant, branch, org, and repo fields are all derived automatically.

For product version bumps (e.g., GitOps v1.18 ŌåÆ v1.19), the branch ref changes via the **filename** ŌĆö rename the file from `v1.18` to `v1.19` in the basename, and `make update` will set `zz_generated_metadata.branch` accordingly.

### 4. Verify Firewatch assignee

Check the [LP Interop - Product QE Contacts](https://docs.google.com/spreadsheets/d/1CBVMbqUsMabSfb-EZ7szijdyC1iH4fKau9rfoW2HTEY/edit?gid=0#gid=0) spreadsheet for the correct `FIREWATCH_DEFAULT_JIRA_ASSIGNEE`.

### 5. Check .config.prowgen

If the config directory has a `.config.prowgen` file, check whether it needs updating to include the new 4.22 job names.

### 6. Generate periodics

```bash
make update
```

This generates/updates the periodics YAML in `ci-operator/jobs/`. Commit both the config and the generated periodics file.

### 7. Create PR

- Target: `openshift/release` main branch
- Title: `<product> ocp 4.22 <type> expansion` (e.g., "acs ocp 4.22 component readiness expansion")
- Body: Include rehearsal job link once available

### 8. Run rehearsal jobs

After PR is created, Prow automatically schedules rehearsal jobs. Wait for them to complete. Add the rehearsal link to the PR body:

```
Rehearsal job link - https://prow.ci.openshift.org/view/gs/test-platform-results/pr-logs/pull/openshift_release/<PR#>/rehearse-<PR#>-periodic-ci-<job-name>/<build-id>
```

### 9. Get review

The PR needs approval from the `cspi-qe-ocp-lp` team in the OWNERS file. Request review from team members.

### 10. Update tracking

- Update the [OCP 4.22 Expansions tracking sheet](https://docs.google.com/spreadsheets/d/1IT8rV8q9T4vZSBh8rFtcSufSq_LUMLDbq6aKCEUNAS8/edit)
- Transition Jira ticket status

## Multi-Test Config Exception

Some products (e.g., Quay) have a single config file with multiple tests, including both FIPS and non-FIPS entries. For these configs:
- Do NOT rename the config file to append `-fips`
- Do NOT append `-fips` to the variant (it would affect all tests in the config)
- Keep `-fips` in the `tests[].as` field for the FIPS test entry only

This is an exception to the standard single-test FIPS naming convention. See `naming-conventions.md` for details.

## Common Pitfalls

- **Do NOT hand-edit periodics YAML** ŌĆö always use `make update`
- **Do NOT manually edit `zz_generated_metadata`** ŌĆö it is auto-generated from the config filename by `make update`
- **Verify branch exists** for products with version bumps (e.g., check `v1.19` branch before referencing it)
- **Cron schedule**: CR jobs use `0 3,15 * * *` (twice daily). FIPS jobs use `0 23 31 2 *` (disabled ŌĆö Feb 31 never occurs). The cron is updated later when the job is ready for production.
- **Test name length**: The `as` field has a 61-char hard limit. The full Job Run Name has a 63-char K8s label limit ŌĆö truncation adds a hash suffix that breaks telemetry. Keep test names concise.
